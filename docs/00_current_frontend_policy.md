# 00. 현재 프론트엔드 정책

## 1. 문서 목적

이 문서는 축구팀 영상분석 플랫폼의 최신 프론트 구현 상태와 경기 영상 편집기 v1의 확정 목표 정책을 요약한다.

새 채팅에서 프론트 작업을 시작할 때 다음 순서로 참고한다.

1. 현재 채팅의 최신 사용자 결정
2. `docs/31_video_editor_v1_requirements.md`
3. 이 문서
4. 기능별 프론트 요구사항 문서
5. GitHub `main`의 실제 프론트·백엔드 소스

현재 로컬 소스와 GitHub `main`은 `video-editor-backend-foundation` 브랜치 생성 시점에 동일하다. 실제 타입, API 주소, Props와 라우트는 GitHub `main`을 직접 확인한다.

---

## 2. 현재 작업 상태

현재 작업은 백엔드 기반 구조 개편 단계다.

```text
브랜치: video-editor-backend-foundation
프론트 수정: 이번 브랜치 범위에서 제외
```

현재 `MatchVideoPage`와 기존 분석 패널은 프론트 전환 전까지 유지한다.

경기 영상 편집기 프론트는 다음 백엔드 기능이 안정화된 뒤 별도 브랜치에서 구현한다.

- 신규 상태와 밀리초 DTO
- DRAFT 저장
- 편집 잠금과 스냅샷
- 렌더링 대기열
- 인증 스트리밍
- 삭제·복구 API

---

## 3. 기술 스택

- React
- TypeScript
- Vite
- Axios
- 반응형 UI

선수 조회 화면은 모바일 우선이다.

COACH와 ANALYST의 영상 업로드·편집 화면은 PC 사용성을 우선한다.

경기 영상 편집기 v1의 공식 편집 지원 대상은 최신 Chrome과 Edge다. 모바일과 태블릿에서는 편집 기능을 제공하지 않는다.

---

## 4. 인증 정책

- 로그인 성공 시 JWT Access Token을 저장한다.
- API 요청은 공통 `axiosInstance`를 사용한다.
- Authorization 헤더를 자동으로 추가한다.
- 새로고침 시 로그인 사용자 정보를 복원한다.
- 401 응답 시 인증 상태를 정리하고 로그인 화면으로 이동한다.
- 파일 재생도 인증과 권한 검증을 거쳐야 한다.

프론트 권한 분기는 UX 목적이며 실제 권한은 백엔드 응답을 기준으로 한다.

## 5. 권한별 UI 정책

### ANALYST

- 일정·공지사항 조회 UI
- 영상 도메인에서는 COACH와 동일한 전체 관리 UI
- 원본 영상 등록·수정·삭제·복구
- 팀·선수 클립 등록·수정·삭제

- 선수 기록 등록과 연결
- 본인 북마크 관리

기존 “ANALYST 삭제 버튼 미노출” 정책은 경기 영상 편집기 v1에서 폐기한다.

### PLAYER

- 스케줄과 공지사항 조회

- 본인 기록과 이벤트 조회

다음 상태의 클립은 목록과 상세에서 노출하지 않는다.

```text
DRAFT
QUEUED
PROCESSING
FAILED
```

## 6. 라우트 정책

라우트 경로는 `ROUTES` 상수로 관리한다.

현재 기존 경로와 `analysisMode`는 새 편집기 전환 전까지 유지한다.

경기 영상 편집기 v1 목표 흐름:

- 경기 영상 상세 또는 편집기 진입
- 하나의 작업공간에서 팀·선수 클립 생성
- 현재 경기의 모든 클립 상태 확인
- DRAFT·FAILED 편집
- 개별·일괄 렌더링
- READY 결과 확인

정확한 신규 경로와 쿼리 파라미터는 백엔드 기반 구조가 확정된 뒤 기존 `routes.ts`, `App.tsx`, `MatchVideoPage.tsx`를 확인해 결정한다.

## 7. API 타입 정책

프론트 타입은 백엔드 DTO와 정확히 일치시킨다.

추측하지 않는 항목:

- API 주소
- 요청·응답 필드명
- 배열 래핑
- 페이지 응답 구조
- Enum
- nullable
- 시간 단위
- 파일 재생 응답 방식

시간 필드는 이름에 단위를 드러낸다.

```text
...Ms → 밀리초
...Sec → 초
```

영상 편집기 v1 신규 canonical 타입은 밀리초를 사용한다. 기존 초 단위 API를 호환하는 동안에는 변환 함수를 중앙화하고 컴포넌트에서 임의 계산하지 않는다.

## 8. 경기 영상 편집기 화면 정책

편집기는 PC 전용이다.

하나의 경기 원본 영상 작업공간에서 다음 기능을 제공한다.

- 원본 영상 재생
- I/O 구간 설정
- 타임라인 핸들
- 1초·5초 이동
- 원본 FPS 기반 프레임 이동
- 선택 구간 반복 재생
- 팀 분석 클립 생성
- 선수 개인 분석 클립 생성
- 드로잉·텍스트
- 화면 정지
- 줌
- 자동 저장 상태
- 편집 잠금 상태
- 복구 스냅샷
- 개별·일괄 렌더링
- 현재 경기의 클립 상태 목록

상태 UI:

```text
DRAFT → 작업 중
QUEUED → 영상 생성 대기 중
PROCESSING → 영상 생성 중
READY → 완료
FAILED → 실패
```

DRAFT가 10일 이상 수정되지 않았으면 `장기 미작업`을 표시한다.

## 9. 저장과 충돌 UI

- 변경 후 1~2초 debounce 저장
- 변경이 계속되면 30초 안전 저장
- 저장 중 중복 요청 방지
- 서버 저장 실패 시 IndexedDB 복구
- 잠금 토큰과 편집 버전을 저장 요청에 포함
- 409 응답 시 저장을 반복하지 않고 읽기 전용 전환 또는 새로고침 안내
- 다른 사용자가 편집 중이면 읽기 전용
- 강제 잠금 해제 후 기존 편집자는 다음 저장 시 충돌 안내

READY 전환 또는 클립 삭제 시 브라우저 로컬 복구 데이터를 삭제한다.

---

## 10. READY 클립 정책

READY 결과 재생에서는 완성 MP4만 사용한다.

드로잉, 텍스트, 화면 정지와 줌은 MP4에 합성돼 있으므로 별도 프론트 오버레이를 중복 렌더링하지 않는다.

READY에서 직접 수정 가능한 항목:

- 제목
- 클립 유형
- 코멘트

직접 수정할 수 없는 항목:

- 원본 영상
- 시작·종료 구간
- 대상 선수
- 드로잉
- 화면 정지
- 줌

영상 내용을 변경할 때는 기존 최종 편집 데이터를 복사해 새 DRAFT를 만든다. 원본 영상이 삭제 상태이거나 실제 파일이 없으면 새 DRAFT 생성을 차단한다.

---

## 11. 영상 재생 URL 정책

현재 `/uploads/**` 상대 URL 결합은 기존 MVP 호환용이다.

경기 영상 편집기 v1 목표:

- `/uploads/**`를 운영 환경에서 직접 공개하지 않는다.
- 인증 스트리밍 또는 짧은 만료 재생 토큰을 사용한다.
- HTTP Range 재생을 지원한다.
- 프론트는 서버 내부 파일 경로를 조합하지 않는다.
- 만료 또는 401·403 응답 시 재인증·권한 안내를 처리한다.
- PLAYER 개인 클립 접근은 대상 선수 본인 여부를 백엔드가 검증한다.

클라우드 저장소 도입 후에는 Signed URL 응답으로 교체할 수 있게 API 사용부를 분리한다.

---

## 12. 클립 삭제 UI

DRAFT, FAILED와 READY에서 영구 삭제를 제공한다.

QUEUED와 PROCESSING에서는 삭제 버튼을 비활성화하거나 노출하지 않는다.

확인 문구:

```text
이 클립을 영구 삭제하시겠습니까?

삭제하면 클립 영상과 모든 관련 데이터가 즉시 영구 삭제되며 복구할 수 없습니다.
```

버튼:

```text
취소
영구 삭제
```

영향 데이터 개수는 표시하지 않는다.

---

## 13. 원본 경기 영상 삭제·복구 UI

원본 삭제 요청 전에 다음만 표시한다.

1. 원본 영상 개수와 제목
2. 원본 북마크 총개수
3. 팀 분석 클립 총개수와 각 제목

경고:

- 원본은 10일간 복구 가능
- 원본 북마크와 팀 분석 클립은 복구되지 않음
- 팀 클립 MP4는 즉시 삭제
- 10일 후 원본 MP4가 영구 삭제되고 복구 불가

`삭제된 영상` 버튼은 별도 페이지가 아니라 모달을 연다.

- 10개 단위 페이지네이션
- `deletedAt DESC`
- 검색과 필터 없음
- 제목과 삭제 일시 표시
- 복구 가능하면 `복구`
- 복구 불가능하면 빨간 표시와 비활성 `복구 불가`

QUEUED 또는 PROCESSING 클립이 존재하면 원본 삭제 요청을 보내지 않고 백엔드 안내를 표시한다.

## 14. 썸네일 정책

- READY 목록은 결과 썸네일을 사용한다.
- 결과 1초 지점 자동 썸네일이 기본이다.
- 썸네일 생성 실패 시 기본 이미지를 표시한다.
- 썸네일 실패만으로 클립 전체를 실패 처리하지 않는다.
- 프론트에서 목록 렌더링을 위해 MP4를 직접 seek해 썸네일을 만들지 않는다.

## 17. React와 TypeScript 규칙

- effect 안에 async 함수를 분리한다.
- 언마운트 후 상태 변경을 방지한다.
- 타입 전용 import는 `import type`을 사용한다.
- 사용하지 않는 변수와 Props를 남기지 않는다.
- Props 추가 시 타입, 구조 분해와 호출부를 함께 수정한다.
- 백엔드에서 제거된 필드를 프론트 타입에서도 제거한다.
- 에러 code, path와 서버 파일 경로를 사용자에게 노출하지 않는다.
- 저장 중 버튼 중복 클릭을 막는다.
- 수정 요청 시 변경을 반영한 전체 파일 코드를 제공한다.
