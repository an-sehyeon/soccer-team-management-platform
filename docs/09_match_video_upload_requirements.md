# 경기 영상 업로드 기능 요구사항

## 1. 결론

경기 영상 업로드 기능은 지도자와 분석관이 실제 동영상 파일과 경기 정보를 등록하고, 지도자/분석관/선수가 업로드된 원본 경기 영상을 조회 및 재생할 수 있게 하는 기능이다.

이번 최종 구현 기준은 다음 정책을 따른다.

* 인증은 이미 구현된 JWT 인증 구조를 사용한다.
* JWT 인증 후 `CustomUserPrincipal`에서 `memberId`, `memberRole`, `isAdmin` 값을 사용한다.
* 지도자 `COACH`는 경기 영상 업로드, 조회, 수정, 삭제가 가능하다.
* 분석관 `ANALYST`는 경기 영상 업로드, 조회, 수정이 가능하다.
* 분석관 `ANALYST`는 경기 영상 삭제가 불가능하다.
* 선수 `PLAYER`는 경기 원본 영상 목록 조회, 상세 조회, 재생이 가능하다.
* 선수 `PLAYER`는 경기 영상 업로드, 수정, 삭제가 불가능하다.
* 관리자 여부 `isAdmin`은 경기 영상 권한의 기본 기준으로 사용하지 않는다.
* 삭제는 실제 DB 삭제가 아니라 `is_deleted = true`로 처리하는 소프트 삭제를 사용한다.
* 초기 버전은 단일 팀 서비스 기준으로 설계한다.
* 현재 DB의 `game_video_upload` 테이블을 우선 사용한다.
* DB에는 영상 파일 자체를 저장하지 않고, 업로드된 파일에 접근할 수 있는 URL을 저장한다.
* 기존 `game_video_upload.url` 컬럼에는 서버가 생성한 영상 접근 URL을 저장한다.
* 경기 영상 등록 API는 `multipart/form-data` 방식으로 동영상 파일과 경기 메타데이터를 함께 받는다.
* 초기 MVP에서는 서버 로컬 디렉터리에 영상 파일을 저장한다.
* 초기 MVP에서는 `.mp4` 파일만 허용한다.
* 초기 MVP에서는 실제 영상 인코딩, 영상 자르기, 썸네일 생성, CDN 연동은 제외한다.
* 초기 MVP에서는 업로드가 완료된 파일을 바로 재생 가능한 상태로 보고 `status = READY`로 저장한다.
* 영상 파일 교체는 이번 수정 API 범위에서 제외한다.
* 경기 영상 수정 API는 경기 제목, 경기일, 장소, 점수, 경기 결과 같은 메타데이터만 수정한다.
* 경기 영상 삭제 시 초기 MVP에서는 DB만 소프트 삭제하고 실제 로컬 파일은 삭제하지 않는다.

## 2. 기능 목적

경기 영상 업로드 기능의 목적은 경기 원본 영상을 플랫폼에 등록하고, 이후 영상 편집과 분석 클립 생성의 기준 데이터를 만드는 것이다.

이 기능은 단순 영상 목록 관리 기능이 아니라, 이후 다음 기능들의 출발점이다.

* 영상 편집기에서 원본 경기 영상 불러오기
* 업로드된 경기 영상 재생
* 팀 분석 클립 생성
* 선수 개인 분석 클립 생성
* 북마크 생성
* 드로잉 데이터 연결
* 경기별 분석 데이터 관리
* 선수 기록과 경기 영상 연결

초기 구현에서는 운영용 대용량 영상 처리보다 실제 업로드와 재생이 가능한 MVP를 우선한다.

따라서 이번 단계에서는 다음을 목표로 한다.

* 사용자가 화면에서 실제 동영상 파일을 선택한다.
* 프론트가 `multipart/form-data`로 서버에 업로드한다.
* 서버가 파일을 로컬 디렉터리에 저장한다.
* 서버가 브라우저에서 접근 가능한 영상 URL을 만든다.
* 서버가 경기 메타데이터와 영상 URL을 DB에 저장한다.
* 프론트가 상세 조회 응답의 `url`을 `<video>` 태그에 연결해 재생한다.

## 3. 사용자 역할

### 3.1 지도자 `COACH`

지도자는 팀 운영과 영상 분석 책임자이므로 경기 영상을 전체 관리할 수 있다.

가능한 기능은 다음과 같다.

* 경기 영상 파일 업로드
* 경기 영상 목록 조회
* 경기 영상 상세 조회
* 경기 영상 재생
* 경기 영상 정보 수정
* 경기 영상 삭제

지도자는 원본 경기 영상을 등록하고 이후 팀 분석 클립과 선수 개인 분석 클립 생성의 기준 영상으로 사용할 수 있다.

### 3.2 분석관 `ANALYST`

분석관은 영상 분석 실무를 담당할 수 있으므로 경기 영상 업로드와 수정은 허용한다.

가능한 기능은 다음과 같다.

* 경기 영상 파일 업로드
* 경기 영상 목록 조회
* 경기 영상 상세 조회
* 경기 영상 재생
* 경기 영상 정보 수정

불가능한 기능은 다음과 같다.

* 경기 영상 삭제

삭제 권한을 막는 이유는 원본 경기 영상이 팀 분석 클립과 선수 개인 분석 클립의 기준 데이터이기 때문이다.

분석관이 실수로 원본 영상을 삭제하면 연결된 분석 데이터 운영에 문제가 생길 수 있다.

### 3.3 선수 `PLAYER`

선수는 등록된 경기 원본 영상을 조회하고 재생할 수 있다.

가능한 기능은 다음과 같다.

* 경기 영상 목록 조회
* 경기 영상 상세 조회
* 경기 원본 영상 재생

불가능한 기능은 다음과 같다.

* 경기 영상 업로드
* 경기 영상 정보 수정
* 경기 영상 삭제

선수에게 원본 영상 조회와 재생은 허용하되, 변경 권한은 주지 않는다.

## 4. 권한 정책

## 4.1 권한 표

| 역할        | 업로드 | 목록 조회 | 상세 조회 | 재생 | 수정 | 삭제 |
| --------- | --: | ----: | ----: | -: | -: | -: |
| `COACH`   |  가능 |    가능 |    가능 | 가능 | 가능 | 가능 |
| `ANALYST` |  가능 |    가능 |    가능 | 가능 | 가능 | 불가 |
| `PLAYER`  |  불가 |    가능 |    가능 | 가능 | 불가 | 불가 |

## 4.2 권한 검증 기준

경기 영상 권한 검증은 반드시 백엔드에서 처리한다.

검증 기준은 다음과 같다.

* 인증되지 않은 사용자는 경기 영상 API에 접근할 수 없다.
* `COACH`, `ANALYST`만 경기 영상을 업로드할 수 있다.
* `COACH`, `ANALYST`, `PLAYER` 모두 경기 영상을 조회할 수 있다.
* `COACH`, `ANALYST`, `PLAYER` 모두 업로드된 경기 영상을 재생할 수 있다.
* `COACH`, `ANALYST`만 경기 영상 정보를 수정할 수 있다.
* `COACH`만 경기 영상을 삭제할 수 있다.
* `PLAYER`가 업로드, 수정, 삭제 API를 직접 호출하면 실패해야 한다.
* `ANALYST`가 삭제 API를 직접 호출하면 실패해야 한다.
* `is_deleted = true`인 영상은 목록, 상세, 수정, 삭제 대상에서 제외한다.
* `isAdmin = true`만으로 경기 영상 업로드, 수정, 삭제 권한을 주지 않는다.

## 4.3 관리자 `isAdmin` 처리

초기 경기 영상 업로드 기능에서는 `isAdmin = true`를 영상 권한 기준으로 사용하지 않는다.

이유는 다음과 같다.

* `isAdmin`은 가입 승인 같은 운영자 권한에 가깝다.
* 영상 관리 권한은 실제 업무 역할인 `COACH`, `ANALYST` 기준으로 판단하는 것이 명확하다.
* 관리자가 선수 역할일 경우 영상 수정/삭제까지 가능해지면 역할 책임이 애매해진다.

단, 추후 운영 관리자에게 전체 영상 관리 권한을 줄 필요가 생기면 별도 정책으로 확장한다.

## 5. 업로드 저장 정책

## 5.1 저장 방식

초기 MVP에서는 서버 로컬 디렉터리에 영상 파일을 저장한다.

예상 저장 위치는 다음과 같다.

```text
backend/uploads/match-videos
```

DB에는 영상 파일 자체를 저장하지 않는다.

DB에는 브라우저가 접근할 수 있는 URL만 저장한다.

```text
game_video_upload.url = /uploads/match-videos/{storedFileName}
```

## 5.2 파일명 정책

사용자가 업로드한 원본 파일명을 그대로 저장하지 않는다.

저장 파일명은 UUID를 기준으로 생성한다.

예시는 다음과 같다.

```text
7c2b9b84-930b-4cc3-bb0b-223c52d9a456.mp4
```

이유는 다음과 같다.

* 한글 파일명, 공백, 특수문자로 인한 경로 문제를 줄일 수 있다.
* 같은 파일명 충돌을 방지할 수 있다.
* 사용자가 올린 원본 파일명을 URL에 노출하지 않을 수 있다.
* 경로 조작 위험을 줄일 수 있다.

## 5.3 허용 파일 형식

초기 MVP에서는 `.mp4`만 허용한다.

검증 기준은 다음과 같다.

* 확장자: `.mp4`
* MIME 타입: `video/mp4`

추후 필요하면 다음 형식으로 확장할 수 있다.

* `.mov`
* `.m4v`
* `.webm`

다만 초기에는 브라우저 기본 재생 호환성과 구현 단순성을 우선하기 때문에 `.mp4`로 제한한다.

## 5.4 파일 크기 제한

초기 개발 테스트 기준으로 파일 크기 제한을 둔다.

권장 초기값은 다음과 같다.

```text
200MB
```

운영 서비스에서는 실제 경기 영상 용량이 훨씬 클 수 있다.

운영 전에는 다음 방식을 다시 검토한다.

* 클라우드 스토리지 직접 업로드
* Presigned URL 업로드
* 멀티파트 업로드
* 업로드 진행률 표시
* 업로드 중단/재시도
* 영상 압축 또는 인코딩 작업 큐

## 5.5 정적 리소스 제공

업로드된 파일은 브라우저에서 재생할 수 있어야 한다.

초기 MVP에서는 Spring MVC 정적 리소스 매핑을 사용한다.

최종 접근 경로는 다음으로 통일한다.

```text
/uploads/match-videos/**
```

예시는 다음과 같다.

```text
http://localhost:8080/uploads/match-videos/7c2b9b84-930b-4cc3-bb0b-223c52d9a456.mp4
```

DB에는 상대 경로 또는 전체 URL 중 하나를 저장할 수 있다.

초기 구현에서는 다음 둘 중 하나를 확정해야 한다.

1. 상대 경로 저장

```text
/uploads/match-videos/{storedFileName}
```

2. 전체 URL 저장

```text
http://localhost:8080/uploads/match-videos/{storedFileName}
```

권장 방식은 상대 경로 저장이다.

이유는 개발 환경과 운영 환경의 도메인이 달라져도 DB 값을 바꾸지 않아도 되기 때문이다.

프론트에서는 상대 경로가 내려오면 `VITE_API_BASE_URL`을 붙여 재생 URL을 만든다.

## 6. 화면 흐름

## 6.1 선수 경기 영상 조회 화면

선수는 모바일에서 경기 영상을 확인할 가능성이 높다.

기본 흐름은 다음과 같다.

1. 선수가 로그인한다.
2. 경기 영상 메뉴에 진입한다.
3. 최신 경기 영상 목록을 확인한다.
4. 경기 영상을 선택한다.
5. 경기 영상 상세 API를 호출한다.
6. 상세 응답의 `url`을 기준으로 영상 재생 화면에서 확인한다.

목록 화면에 표시할 정보는 다음과 같다.

* 경기 영상 ID
* 경기 제목
* 경기 날짜
* 장소
* 홈 점수
* 원정 점수
* 경기 결과
* 영상 상태
* 등록일시

상세 화면에 표시할 정보는 다음과 같다.

* 경기 영상 ID
* 영상 URL
* 경기 제목
* 경기 날짜
* 장소
* 홈 점수
* 원정 점수
* 경기 결과
* 영상 상태
* 등록자 ID
* 등록자 이름
* 생성일시
* 수정일시

## 6.2 지도자/분석관 경기 영상 관리 화면

지도자와 분석관은 PC에서 영상 업로드와 편집을 시작할 가능성이 높다.

기본 흐름은 다음과 같다.

1. 지도자 또는 분석관이 경기 영상 관리 화면에 진입한다.
2. 경기 영상 등록 버튼 또는 등록 폼을 확인한다.
3. 동영상 파일을 선택한다.
4. 경기 제목, 경기 날짜, 장소, 홈 점수, 원정 점수, 경기 결과를 입력한다.
5. 저장 버튼을 누른다.
6. 프론트는 파일과 메타데이터를 `multipart/form-data`로 전송한다.
7. 백엔드는 파일을 서버 로컬 디렉터리에 저장한다.
8. 백엔드는 저장된 파일 URL과 경기 정보를 DB에 저장한다.
9. 저장 성공 후 경기 영상 목록을 갱신한다.
10. 등록된 영상을 선택해 상세 정보와 영상을 확인한다.
11. 필요한 경우 경기 정보를 수정한다.
12. 지도자는 삭제할 수 있고, 분석관은 삭제할 수 없다.

## 6.3 프론트 버튼 노출 정책

* `COACH`에게 업로드, 수정, 삭제 버튼을 보여준다.
* `ANALYST`에게 업로드, 수정 버튼만 보여준다.
* `PLAYER`에게 조회 및 재생 화면만 보여준다.
* 버튼 숨김은 UX 목적이며, 실제 권한 차단은 백엔드가 담당한다.

## 7. API 흐름

## 7.1 API 설계 방향

경기 영상 조회 API는 지도자, 분석관, 선수가 모두 사용하므로 공통 API로 둔다.

경기 영상 업로드/수정 API는 지도자와 분석관이 사용할 수 있으므로 관리 API로 둔다.

경기 영상 삭제 API는 지도자만 가능하게 서비스 계층에서 검증한다.

최종 API는 다음과 같다.

```http
GET    /api/match-videos
GET    /api/match-videos/{matchVideoId}
POST   /api/management/match-videos
PATCH  /api/management/match-videos/{matchVideoId}
DELETE /api/coach/match-videos/{matchVideoId}
GET    /uploads/match-videos/{storedFileName}
```

`management` 경로는 `COACH`, `ANALYST`가 함께 사용하는 관리성 API라는 의미다.

삭제는 정책상 `COACH`만 가능하므로 `/api/coach` 경로로 분리한다.

---

## 7.2 경기 영상 목록 조회

```http
GET /api/match-videos?page=0&size=20
```

사용 가능 역할은 다음과 같다.

* `COACH`
* `ANALYST`

## 7.4 경기 영상 파일 업로드 및 등록

```http
POST /api/management/match-videos
Content-Type: multipart/form-data
```

사용 가능 역할은 다음과 같다.

* `COACH`
* `ANALYST`

요청 파트는 다음과 같다.

| 파트명           | 타입     | 필수 여부 | 설명                    |
| ------------- | ------ | ----: | --------------------- |
| `videoFile`   | File   |    필수 | 업로드할 경기 영상 파일         |
| `title`       | String |    필수 | 경기 제목                 |
| `gameDate`    | String |    필수 | 경기 날짜와 시간             |
| `place`       | String |    필수 | 경기 장소                 |
| `homeScore`   | Number |    필수 | 홈팀 점수                 |
| `awayScore`   | Number |    필수 | 원정팀 점수                |
| `matchResult` | String |    필수 | `WIN`, `DRAW`, `LOSS` |

요청 예시는 다음과 같다.

```text
videoFile: match-test.mp4
title: 2026 춘계리그 vs 서울FC
gameDate: 2026-06-20T15:00:00
place: 메인 운동장
homeScore: 2
awayScore: 1
matchResult: WIN
```

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 로그인 회원의 역할이 `COACH` 또는 `ANALYST`인지 확인한다.
3. `videoFile`이 비어 있는지 확인한다.
4. 파일 확장자가 `.mp4`인지 확인한다.
5. MIME 타입이 `video/mp4`인지 확인한다.
6. 파일 크기가 허용 범위 안인지 확인한다.
7. 경기 제목, 경기 날짜, 장소, 점수, 경기 결과를 검증한다.
8. 점수 값이 음수가 아닌지 검증한다.
9. 경기 결과 값이 허용된 Enum인지 검증한다.
10. UUID 기반 저장 파일명을 생성한다.
11. 서버 로컬 업로드 디렉터리에 파일을 저장한다.
12. 저장된 파일 접근 URL을 생성한다.
13. `member_id`에 업로드한 회원 ID를 저장한다.
14. `url`에 생성된 영상 접근 URL을 저장한다.
15. `status`는 `READY`로 저장한다.
16. 경기 영상 정보를 저장한다.
17. 생성된 경기 영상 ID를 반환한다.

응답 예시는 다음과 같다.

```json
{
  "matchVideoId": 1,
  "message": "경기 영상이 업로드되었습니다."
}
```

## 7.7 업로드된 영상 파일 조회

```http
GET /uploads/match-videos/{storedFileName}
```

사용 목적은 브라우저 `<video>` 태그에서 업로드된 영상을 재생하는 것이다.

초기 구현 기준은 다음과 같다.

* 업로드된 파일은 서버 로컬 디렉터리에 저장한다.
* Spring MVC 정적 리소스 설정으로 `/uploads/match-videos/**` 경로를 제공한다.
* 상세 조회 응답의 `url`은 이 정적 리소스 경로를 포함한다.
* 프론트는 `<video src={videoUrl} controls>` 방식으로 재생한다.

주의사항은 다음과 같다.

* 이 방식은 개발 및 MVP 검증에 적합하다.
* 운영 서비스에서는 원본 영상 접근 권한, URL 노출, 저장소 비용, CDN, 서명 URL을 별도로 검토해야 한다.
* `<video>` 태그는 Authorization 헤더를 쉽게 붙이기 어렵기 때문에, 운영 단계에서는 공개 URL 또는 서명 URL 방식이 필요하다.

## 8.3 신규 컬럼 추가 여부

이번 MVP에서는 신규 DB 컬럼을 추가하지 않는다.

추가하지 않는 컬럼은 다음과 같다.

* 원본 파일명
* 저장 파일명
* 파일 크기
* MIME 타입
* 영상 길이
* 썸네일 URL
* 저장소 종류
* 업로드 실패 사유

이유는 다음과 같다.

* 현재 DB에는 `url` 컬럼이 이미 있다.
* 이번 목표는 실제 파일 업로드와 재생 검증이다.
* 과도한 DB 변경 없이 기존 구조를 활용할 수 있다.
* 운영 단계에서 스토리지 구조가 바뀔 수 있으므로 파일 메타데이터 컬럼은 추후 재검토한다.

단, 운영 서비스 전에는 위 컬럼 추가를 적극 검토한다.

특히 팀 분석 클립과 선수 개인 분석 클립의 시간 검증을 위해 추후 `duration_sec` 컬럼은 필요할 수 있다.

8. 기존 경기 영상 등록 API를 `multipart/form-data` 방식으로 변경할 범위 확인

## 10.2 백엔드 2단계: 업로드 설정 추가

1. `application.properties`에 multipart 업로드 크기 제한 추가
2. 업로드 디렉터리 경로 설정 추가
3. `/uploads/match-videos/**` 정적 리소스 매핑 설정 추가

## 10.3 백엔드 3단계: 파일 저장 서비스 작성

1. 경기 영상 파일 저장 전용 서비스 작성
2. 파일 미첨부 여부 검증
3. 빈 파일 여부 검증
4. `.mp4` 확장자 검증
5. `video/mp4` MIME 타입 검증
6. 파일 크기 검증
7. UUID 기반 저장 파일명 생성
8. 서버 로컬 디렉터리에 파일 저장
9. 브라우저 접근 URL 반환

## 10.4 백엔드 4단계: DTO 작성 및 수정

1. `CreateMatchVideoRequestDTO`는 기존 JSON 등록 방식에서 파일 업로드 메타데이터용으로 사용 범위를 조정한다.
2. 업로드 파일은 DTO에 넣지 않고 Controller에서 `MultipartFile`로 받는 방식을 우선 검토한다.
3. `UpdateMatchVideoRequestDTO`는 파일 교체 없이 메타데이터 수정 전용으로 유지한다.
4. `CreateMatchVideoResponseDTO` 작성 또는 유지
5. `UpdateMatchVideoResponseDTO` 작성 또는 기존 상세 응답 재사용
6. `MatchVideoListResponseDTO` 유지
7. `MatchVideoDetailResponseDTO` 유지
8. `MatchVideoPageResponseDTO` 유지

## 10.5 백엔드 5단계: 권한 검증 로직 작성

1. `CustomUserPrincipal`에서 `memberId`, `memberRole`, `isAdmin` 사용
2. 경기 영상 업로드/수정 권한 검증 메서드 작성
3. `COACH`, `ANALYST`가 아니면 업로드/수정 실패 처리
4. 경기 영상 삭제 권한 검증 메서드 작성
5. `COACH`가 아니면 삭제 실패 처리
6. 조회는 `COACH`, `ANALYST`, `PLAYER` 허용
7. `isAdmin`만으로 영상 변경 권한을 주지 않도록 처리

## 10.6 백엔드 6단계: Service 작성 및 수정

1. 경기 영상 파일 업로드 및 등록 기능 작성
2. 업로드 파일 저장 후 생성된 URL을 DB에 저장
3. `status`는 업로드 완료 후 `READY`로 저장
4. 경기 영상 목록 조회 기능 유지
5. 경기 영상 상세 조회 기능 유지
6. 경기 영상 수정 기능은 메타데이터 수정만 처리
7. 경기 영상 삭제 기능은 소프트 삭제 처리
8. 삭제 시 실제 파일 삭제는 이번 MVP에서 제외

* `duration_sec` 컬럼 추가
* 원본 파일명 저장
* 저장 파일명 저장
* 파일 크기 저장

## 12. 주의사항

* 경기 영상 업로드, 수정, 삭제 권한은 프론트가 아니라 백엔드에서 반드시 검증한다.
* 선수는 경기 원본 영상 조회와 재생은 가능하지만 변경 권한은 없다.
* 분석관은 업로드와 수정은 가능하지만 삭제는 불가능하다.
* 삭제는 실제 삭제가 아니라 `is_deleted = true`로 처리한다.
* 초기 MVP에서는 삭제 시 실제 영상 파일은 삭제하지 않는다.
* 조회 API는 삭제되지 않은 경기 영상만 반환한다.
* 이번 구현에서는 실제 파일 업로드와 재생을 포함한다.
* 이번 구현에서는 영상 인코딩, 썸네일 생성, CDN 연동은 제외한다.
* 이번 구현에서는 경기 영상 수정 시 파일 교체를 제외한다.
* DB에는 영상 파일 자체를 저장하지 않고 URL만 저장한다.
* 업로드 파일명은 원본 파일명을 그대로 사용하지 않고 UUID 기반으로 저장한다.
* 초기 MVP에서는 `.mp4`만 허용한다.
* 파일 확장자뿐 아니라 MIME 타입도 검증한다.
* 대용량 경기 영상은 서버 디스크 용량과 업로드 시간 문제가 생길 수 있다.
* 로컬 파일 저장 방식은 개발 및 MVP 검증에는 적합하지만 운영 서비스에서는 클라우드 스토리지와 CDN을 다시 설계해야 한다.
* `<video>` 태그는 일반적으로 Authorization 헤더를 붙여 요청하기 어렵기 때문에 운영에서는 공개 URL, 서명 URL, 스트리밍 서버 방식을 검토해야 한다.
* 로컬 업로드 디렉터리는 Git에 포함하지 않는다.
* 현재 서비스는 단일 팀 기준이므로 `team_id`를 사용하지 않는다.
* 추후 여러 팀을 받을 경우 `game_video_upload` 테이블에 `team_id` 또는 팀 소속 구조가 필요하다.
* 원본 경기 영상은 이후 팀 분석 클립과 선수 개인 분석 클립의 기준 데이터이므로 삭제 정책을 신중하게 유지해야 한다.
