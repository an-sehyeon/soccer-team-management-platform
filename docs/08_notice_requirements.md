# 공지사항 관리 기능 요구사항

## 1. 결론

공지사항 관리 기능은 지도자가 팀 공지를 등록, 수정, 삭제하고 선수와 분석관이 공지를 조회하는 기능이다.

초기 구현은 다음 정책을 기준으로 진행한다.

- 인증은 이미 구현된 JWT 인증 구조를 사용한다.
- JWT 인증 후 `CustomUserPrincipal`에서 `memberId`, `memberRole`, `isAdmin` 값을 사용한다.
- 지도자 `COACH`는 공지사항 등록, 수정, 삭제, 조회가 가능하다.
- 선수 `PLAYER`는 공지사항 조회만 가능하다.
- 분석관 `ANALYST`는 공지사항 조회만 가능하다.
- 관리자 여부 `isAdmin`은 공지사항 권한의 기본 기준으로 사용하지 않는다.
- 삭제는 실제 DB 삭제가 아니라 `is_deleted = true`로 처리하는 소프트 삭제를 사용한다.
- 초기 버전은 단일 팀 서비스 기준으로 설계한다.
- 현재 DB의 `notice` 테이블 구조를 우선 사용하고, 별도 팀 컬럼은 추가하지 않는다.
- 공지 목록은 최신순으로 조회하되, 중요 공지는 상단에 우선 노출할 수 있도록 정렬한다.

---

## 2. 기능 목적

공지사항 관리 기능의 목적은 지도자가 훈련 변경, 경기 준비, 소집 시간, 준비물, 전달 사항 등을 팀 구성원에게 공유할 수 있게 하는 것이다.

이 기능은 선수들이 모바일에서 자주 확인하는 운영 기능이므로 다음 기준을 우선한다.

- 선수는 최신 공지와 중요 공지를 빠르게 확인할 수 있어야 한다.
- 지도자는 공지 제목, 내용, 중요 여부를 쉽게 등록할 수 있어야 한다.
- 권한 없는 사용자가 공지를 등록, 수정, 삭제할 수 없어야 한다.
- 삭제된 공지는 목록과 상세 조회에 노출되지 않아야 한다.
- 추후 공지 읽음 여부, 푸시 알림, 첨부파일 기능으로 확장할 수 있어야 한다.

## 3. 사용자 역할

### 3.1 지도자 `COACH`

지도자는 팀 운영 책임자이므로 공지사항을 관리할 수 있다.

가능한 기능은 다음과 같다.

- 공지사항 등록
- 공지사항 수정
- 공지사항 삭제
- 공지사항 목록 조회
- 공지사항 상세 조회

### 3.2 선수 `PLAYER`

선수는 지도자가 등록한 공지사항을 확인하는 사용자이다.

가능한 기능은 다음과 같다.

- 공지사항 목록 조회
- 공지사항 상세 조회

불가능한 기능은 다음과 같다.

- 공지사항 등록
- 공지사항 수정
- 공지사항 삭제

### 3.3 분석관 `ANALYST`

분석관은 영상 분석 업무를 담당하지만 팀 공지 관리 권한은 가지지 않는다.

가능한 기능은 다음과 같다.

- 공지사항 목록 조회
- 공지사항 상세 조회

불가능한 기능은 다음과 같다.

- 공지사항 등록
- 공지사항 수정
- 공지사항 삭제

## 4. 권한 정책

## 4.1 권한 표

| 역할 | 등록 | 수정 | 삭제 | 목록 조회 | 상세 조회 |
|---|---:|---:|---:|---:|---:|
| `COACH` | 가능 | 가능 | 가능 | 가능 | 가능 |
| `PLAYER` | 불가 | 불가 | 불가 | 가능 | 가능 |
| `ANALYST` | 불가 | 불가 | 불가 | 가능 | 가능 |

## 4.2 권한 검증 기준

공지사항 권한 검증은 반드시 백엔드에서 처리한다.

프론트에서 버튼을 숨기더라도 API를 직접 호출할 수 있으므로 서비스 계층 또는 공통 권한 검증 메서드에서 다시 확인한다.

검증 기준은 다음과 같다.

- 인증되지 않은 사용자는 모든 공지사항 API에 접근할 수 없다.
- `memberRole = COACH`인 회원만 등록, 수정, 삭제할 수 있다.
- `memberRole = PLAYER`인 회원은 조회만 가능하다.
- `memberRole = ANALYST`인 회원은 조회만 가능하다.
- 삭제된 공지사항은 수정, 삭제, 상세 조회 대상이 될 수 없다.

## 4.3 관리자 `isAdmin` 처리

초기 공지사항 기능에서는 `isAdmin = true`만으로 공지사항 등록, 수정, 삭제 권한을 주지 않는다.

이유는 다음과 같다.

- `isAdmin`은 가입 승인 같은 운영자 권한에 가깝다.
- 공지사항 변경 권한은 팀 운영 역할인 `COACH` 기준으로 판단하는 것이 명확하다.
- 관리자가 분석관 역할일 경우 공지사항 수정까지 허용하면 역할 책임이 애매해질 수 있다.

단, 추후 운영 관리자에게 전체 공지 관리 권한을 줄 필요가 생기면 별도 정책으로 확장한다.

## 4.4 작성자 기준 수정/삭제 정책

초기 버전에서는 `COACH`라면 본인이 작성하지 않은 공지사항도 수정, 삭제할 수 있게 한다.

이유는 다음과 같다.

- 현재 서비스는 단일 팀 기준이다.
- 지도자는 팀 운영 책임자로 보고 공지 관리 권한을 넓게 둔다.
- 공지 작성자가 부재중일 때도 다른 지도자가 수정할 수 있어야 한다.

운영 중 이력이 중요해지면 다음 기능을 추가한다.

- 작성자 본인만 수정 가능
- 관리자 또는 감독만 삭제 가능
- 공지 수정 이력 테이블
- 마지막 수정자 컬럼

## 5. 화면 흐름

현재 단계에서는 프론트를 구현하지 않지만, 백엔드 API 응답은 추후 화면을 고려해 설계한다.

## 5.1 선수/분석관 공지사항 조회 화면

선수와 분석관은 모바일 조회 사용성을 우선한다.

기본 화면 흐름은 다음과 같다.

1. 사용자가 로그인한다.
2. 메인 화면 또는 공지사항 메뉴에 진입한다.
3. 중요 공지와 최신 공지를 먼저 확인한다.
4. 공지사항 항목을 누르면 상세 내용을 확인한다.

목록 화면에 표시할 정보는 다음과 같다.

- 공지사항 ID
- 제목
- 중요 공지 여부
- 작성자 이름
- 작성일시
- 수정일시

상세 화면에 표시할 정보는 다음과 같다.

- 제목
- 내용
- 중요 공지 여부
- 작성자 이름
- 작성일시
- 수정일시

## 5.2 지도자 공지사항 관리 화면

지도자는 PC와 모바일 모두 사용할 수 있지만, 관리 화면은 입력 편의성을 우선한다.

기본 화면 흐름은 다음과 같다.

1. 지도자가 공지사항 관리 화면에 진입한다.
2. 공지사항 목록을 확인한다.
3. 등록 버튼을 누른다.
4. 제목, 내용, 중요 여부를 입력한다.
5. 저장 버튼을 누른다.
6. 저장 성공 후 목록을 갱신한다.
7. 기존 공지사항을 선택하면 수정 또는 삭제할 수 있다.

## 5.3 프론트 버튼 노출 정책

- `COACH`에게만 등록, 수정, 삭제 버튼을 보여준다.
- `PLAYER`, `ANALYST`에게는 조회 화면만 보여준다.
- 버튼 숨김은 UX 목적이며, 실제 권한 차단은 백엔드가 담당한다.

## 6. API 흐름

## 6.1 API 설계 방향

공지사항 조회 API는 지도자, 선수, 분석관이 모두 사용하므로 공통 API로 둔다.

공지사항 등록, 수정, 삭제 API는 지도자만 사용할 수 있으므로 지도자 전용 경로로 분리한다.

권장 API 초안은 다음과 같다.

```http
GET    /api/notices
GET    /api/notices/{noticeId}
POST   /api/coach/notices
PATCH  /api/coach/notices/{noticeId}
DELETE /api/coach/notices/{noticeId}
```

## 6.2 공지사항 목록 조회

```http
GET /api/notices?page=0&size=20&importantOnly=false
```

사용 가능 역할은 다음과 같다.

- `COACH`
- `PLAYER`
- `ANALYST`

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 로그인 회원의 역할이 `COACH`, `PLAYER`, `ANALYST` 중 하나인지 확인한다.
3. 페이지 번호와 페이지 크기를 검증한다.
4. `importantOnly = true`이면 중요 공지만 조회한다.
5. `is_deleted = false`인 공지사항만 조회한다.
6. `is_important DESC`, `created_at DESC` 기준으로 정렬한다.
7. 목록 응답을 반환한다.

응답 예시는 다음과 같다.

```json
{
  "notices": [
    {
      "noticeId": 1,
      "title": "토요일 연습경기 소집 안내",
      "isImportant": true,
      "writerId": 3,
      "writerName": "김지도",
      "createdAt": "2026-06-20T10:00:00",
      "updatedAt": "2026-06-20T10:00:00"
    }
  ],
  "page": 0,
  "size": 20,
  "totalElements": 1,
  "totalPages": 1
}
```

## 6.3 공지사항 상세 조회

```http
GET /api/notices/{noticeId}
```

사용 가능 역할은 다음과 같다.

- `COACH`
- `PLAYER`
- `ANALYST`

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 공지사항 ID로 공지사항을 조회한다.
3. 공지사항이 존재하지 않거나 삭제된 경우 실패 처리한다.
4. 상세 정보를 반환한다.

응답 예시는 다음과 같다.

```json
{
  "noticeId": 1,
  "title": "토요일 연습경기 소집 안내",
  "content": "토요일 오전 9시까지 메인 운동장으로 집합합니다.",
  "isImportant": true,
  "writerId": 3,
  "writerName": "김지도",
  "createdAt": "2026-06-20T10:00:00",
  "updatedAt": "2026-06-20T10:00:00"
}
```

## 6.4 공지사항 등록

```http
POST /api/coach/notices
```

사용 가능 역할은 다음과 같다.

- `COACH`

요청 예시는 다음과 같다.

```json
{
  "title": "토요일 연습경기 소집 안내",
  "content": "토요일 오전 9시까지 메인 운동장으로 집합합니다.",
  "isImportant": true
}
```

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 로그인 회원의 역할이 `COACH`인지 확인한다.
3. 요청 필수값을 검증한다.
4. `member_id`에 등록한 지도자의 회원 ID를 저장한다.
5. 공지사항을 저장한다.
6. 생성된 공지사항 ID를 반환한다.

## 6.5 공지사항 수정

```http
PATCH /api/coach/notices/{noticeId}
```

사용 가능 역할은 다음과 같다.

- `COACH`

요청 예시는 다음과 같다.

```json
{
  "title": "토요일 연습경기 소집 시간 변경",
  "content": "토요일 오전 8시 30분까지 메인 운동장으로 집합합니다.",
  "isImportant": true
}
```

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 로그인 회원의 역할이 `COACH`인지 확인한다.
3. 공지사항 ID로 공지사항을 조회한다.
4. 공지사항이 존재하지 않거나 삭제된 경우 실패 처리한다.
5. 요청 값을 검증한다.
6. 공지사항 정보를 수정한다.
7. 수정된 공지사항 정보를 반환한다.

## 6.6 공지사항 삭제

```http
DELETE /api/coach/notices/{noticeId}
```

사용 가능 역할은 다음과 같다.

- `COACH`

처리 흐름은 다음과 같다.

1. JWT 인증 정보를 확인한다.
2. 로그인 회원의 역할이 `COACH`인지 확인한다.
3. 공지사항 ID로 공지사항을 조회한다.
4. 공지사항이 존재하지 않거나 이미 삭제된 경우 실패 처리한다.
5. `is_deleted = true`로 변경한다.
6. 삭제 완료 응답을 반환한다.

## 7. DB 설계 방향

## 7.1 현재 사용할 테이블

초기 구현에서는 현재 DB의 `notice` 테이블을 그대로 사용한다.

| 컬럼 | 사용 목적 |
|---|---|
| `id` | 공지사항 PK |
| `member_id` | 공지사항 작성 회원 ID |
| `title` | 공지사항 제목 |
| `content` | 공지사항 내용 |
| `is_important` | 중요 공지 여부 |
| `is_deleted` | 소프트 삭제 여부 |
| `created_at` | 생성일시 |
| `updated_at` | 수정일시 |

## 7.2 Entity 매핑 방향

`notice` 테이블은 `NoticeEntity`로 매핑한다.

권장 필드명은 다음과 같다.

| DB 컬럼 | Entity 필드명 |
|---|---|
| `id` | `id` |
| `member_id` | `member` |
| `title` | `title` |
| `content` | `content` |
| `is_important` | `isImportant` |
| `is_deleted` | `isDeleted` |
| `created_at` | `createdAt` |
| `updated_at` | `updatedAt` |

## 7.3 DB 변경 여부

초기 구현에서는 DB 컬럼을 추가하지 않는다.

현재 `notice` 테이블만으로 다음 기능을 처리할 수 있다.

- 공지사항 등록
- 공지사항 수정
- 공지사항 삭제
- 중요 공지 표시
- 작성자 연결
- 삭제 여부 관리

추후 운영에서 필요해지면 다음 컬럼 또는 테이블을 검토한다.

- 조회수
- 공지 읽음 여부 테이블
- 첨부파일 테이블
- 공지 수정 이력 테이블
- 마지막 수정자 컬럼
- 공지 고정 기간 컬럼

- `isImportant`

필드 방향은 다음과 같다.

- `title`
- `content`
- `isImportant`

`NoticeListResponseDTO`

필드 방향은 다음과 같다.

- `noticeId`
- `title`
- `isImportant`
- `writerId`
- `writerName`
- `createdAt`
- `updatedAt`

## 8.4 상세 응답 DTO

`NoticeDetailResponseDTO`

필드 방향은 다음과 같다.

- `noticeId`
- `title`
- `content`
- `isImportant`
- `writerId`
- `writerName`
- `createdAt`
- `updatedAt`

필드 방향은 다음과 같다.

- `notices`
- `page`
- `size`
- `totalElements`
- `totalPages`

| 상황 | 처리 방식 | 권장 HTTP 상태 |
|---|---|---|
| JWT 없음 | 인증 실패 | `401 Unauthorized` |
| JWT 만료 | 인증 실패 | `401 Unauthorized` |
| 선수의 등록 API 호출 | 접근 차단 | `403 Forbidden` |
| 선수의 수정 API 호출 | 접근 차단 | `403 Forbidden` |
| 선수의 삭제 API 호출 | 접근 차단 | `403 Forbidden` |
| 분석관의 등록 API 호출 | 접근 차단 | `403 Forbidden` |
| 분석관의 수정 API 호출 | 접근 차단 | `403 Forbidden` |
| 분석관의 삭제 API 호출 | 접근 차단 | `403 Forbidden` |
| 존재하지 않는 공지사항 조회 | 조회 실패 | `404 Not Found` |

| 페이지 번호가 음수 | 요청 실패 | `400 Bad Request` |
| 페이지 크기가 허용 범위 초과 | 요청 실패 | `400 Bad Request` |

## 10. 구현 순서

## 10.1 백엔드 1단계: 기본 구조 확인

1. `NoticeEntity` 필드와 DB 컬럼 매핑 확인
2. `MemberEntity`와 `NoticeEntity` 연관관계 확인
3. `NoticeRepository` 생성
4. 기존 공통 예외 구조와 에러 코드 확인
5. 기존 스케줄 권한 검증 방식 참고

## 10.2 백엔드 2단계: DTO 작성

1. `NoticeCreateRequestDTO` 작성
2. `NoticeUpdateRequestDTO` 작성
3. `NoticeListResponseDTO` 작성
4. `NoticeDetailResponseDTO` 작성
5. `NoticePageResponseDTO` 작성

## 10.3 백엔드 3단계: 권한 검증 로직 작성

1. `CustomUserPrincipal`에서 `memberId`, `memberRole`, `isAdmin` 사용

5. `isAdmin`만으로 공지사항 변경 권한을 주지 않도록 처리

## 10.4 백엔드 4단계: Service 작성

## 12. 주의사항

- 공지사항 등록, 수정, 삭제 권한은 프론트가 아니라 백엔드에서 반드시 검증한다.
- 분석관은 조회만 가능하므로 영상 관리 권한과 공지사항 관리 권한을 혼동하지 않는다.
- 삭제는 실제 삭제가 아니라 `is_deleted = true`로 처리한다.
- 조회 API는 삭제되지 않은 공지사항만 반환한다.
- 중요 공지는 일반 공지보다 먼저 노출될 수 있도록 정렬한다.
- 공지 내용은 `TEXT` 타입이지만, 운영 안정성을 위해 요청 크기 제한을 둘 필요가 있다.
- 현재 서비스는 단일 팀 기준이므로 `team_id`를 사용하지 않는다.
- 추후 여러 팀을 받을 경우 `notice` 테이블에 `team_id` 또는 팀 소속 구조가 필요하다.
