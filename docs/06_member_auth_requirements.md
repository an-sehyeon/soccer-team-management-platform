# 회원가입/로그인 기능 요구사항

## 1. 결론

회원가입과 로그인 기능은 단순히 계정을 만드는 기능이 아니라, 플랫폼 전체 권한 제어의 시작점이다.

초기 구현은 다음 정책을 기준으로 진행한다.

* 사용자는 `COACH`, `PLAYER`, `ANALYST` 역할로 가입할 수 있다.
* 신규 가입자는 기본적으로 `PENDING` 상태로 저장한다.
* `APPROVED` 상태가 된 회원만 로그인할 수 있다.
* 비밀번호는 반드시 BCrypt로 암호화해서 저장한다.
* 로그인 성공 시 JWT Access Token을 발급한다.
* JWT에는 권한 검증에 필요한 최소 정보만 담는다.
* 실제 기능 접근 권한은 프론트가 아니라 Spring Security와 백엔드 서비스에서 검증한다.
* 회원가입 화면은 역할 선택에 따라 입력 항목이 달라진다.
* 선수는 학년과 등번호를 필수로 입력해야 한다.
* 선수 학년은 1~4 사이만 허용한다.
* 선수 등번호는 기존 선수와 중복될 수 없다.
* 지도자와 분석관은 학년, 출신학교, 등번호를 필수로 입력하지 않는다.
* 관리자는 승인 대기 회원을 단일 또는 일괄 승인/거절할 수 있다.

---

## 2. 회원가입 목적

회원가입 기능의 목적은 지도자, 선수, 분석관이 플랫폼에 접근할 수 있는 계정을 생성하는 것이다.

이 플랫폼에서는 이후 모든 기능이 로그인 회원의 역할과 승인 상태를 기준으로 동작한다.

따라서 회원가입 기능은 다음 책임을 가진다.

* 회원의 기본 정보 등록
* 로그인 ID 중복 방지
* 비밀번호 안전 저장
* 지도자/선수/분석관 역할 구분
* 역할별 입력 항목 제어
* 선수 학년/등번호 데이터 정합성 검증
* 승인 전 사용자의 서비스 접근 차단
* 추후 영상, 스케줄, 공지사항, 기록 기능의 권한 검증 기준 제공

회원가입은 단순 입력 저장이 아니라, 서비스 접근 권한을 만들기 전 단계로 봐야 한다.

## 3. 가입 가능한 역할

### 3.1 1차 가입 허용 역할

초기 버전에서 가입 가능한 역할은 다음 3개로 제한한다.

| 역할  | DB 값      | 설명                                           | 가입 허용 여부 |
| --- | --------- | -------------------------------------------- | -------- |
| 지도자 | `COACH`   | 스케줄, 공지사항, 영상 업로드, 영상 편집, 선수 기록 관리 가능        | 허용       |
| 선수  | `PLAYER`  | 스케줄, 공지사항, 팀 분석 영상, 본인 개인 분석 영상, 본인 기록 조회 가능 | 허용       |
| 분석관 | `ANALYST` | 영상 업로드, 영상 편집, 분석 클립 관리 가능                   | 허용       |

### 3.2 1차 가입에서 제외할 역할

| 역할  | DB 값              | 처리 방식                                             |
| --- | ----------------- | ------------------------------------------------- |
| 관리자 | `is_admin = true` | 일반 회원가입으로 생성하지 않음. 최초 관리자 계정은 수동 등록 또는 초기 데이터로 생성 |

### 3.3 역할별 입력 정보

회원가입 화면에서는 역할을 라디오 버튼 또는 선택 버튼 형태로 먼저 선택한다.

역할 선택에 따라 입력 칸을 다르게 보여준다.

#### 공통 입력

모든 역할이 반드시 입력해야 하는 정보는 다음과 같다.

* 로그인 ID
* 비밀번호
* 이름
* 휴대폰 번호
* 역할

#### 선수 선택 시 입력 항목

선수는 팀 내 개인 식별과 기록 관리를 위해 추가 정보를 입력해야 한다.

| 항목   | 필수 여부 | 설명           |
| ---- | ----- | ------------ |
| 학년   | 필수    | 1~4 사이 값만 허용 |
| 등번호  | 필수    | 기존 선수와 중복 불가 |
| 출신학교 | 선택    | 선수 출신학교      |

#### 지도자 선택 시 입력 항목

지도자는 선수 정보가 아니므로 선수용 입력 항목을 필수로 받지 않는다.

| 항목   | 필수 여부   | 설명      |
| ---- | ------- | ------- |
| 학년   | 입력하지 않음 | 화면에서 숨김 |
| 등번호  | 입력하지 않음 | 화면에서 숨김 |
| 출신학교 | 입력하지 않음 | 화면에서 숨김 |

#### 분석관 선택 시 입력 항목

분석관도 선수 정보가 아니므로 선수용 입력 항목을 필수로 받지 않는다.

| 항목   | 필수 여부   | 설명      |
| ---- | ------- | ------- |
| 학년   | 입력하지 않음 | 화면에서 숨김 |
| 등번호  | 입력하지 않음 | 화면에서 숨김 |
| 출신학교 | 입력하지 않음 | 화면에서 숨김 |

### 3.4 회원가입 화면 입력 흐름

회원가입 화면의 기본 흐름은 다음과 같다.

1. 사용자가 역할을 선택한다.
2. 역할이 `PLAYER`이면 선수 추가 입력 영역을 보여준다.
3. 역할이 `COACH` 또는 `ANALYST`이면 선수 추가 입력 영역을 숨긴다.
4. 사용자가 회원가입 버튼을 누른다.
5. 프론트에서 1차 필수값 검증을 한다.
6. 백엔드에서 다시 역할별 필수값을 검증한다.
7. 검증이 통과되면 `PENDING` 상태로 가입 신청을 저장한다.

### 3.5 역할별 입력 검증 정책

역할별 입력 검증은 프론트와 백엔드에서 모두 처리한다.

프론트 검증은 사용자 편의를 위한 것이고, 실제 보안과 데이터 정합성은 백엔드 검증이 책임진다.

백엔드 검증 기준은 다음과 같다.

| 역할        | 학년          | 등번호       | 출신학교  |
| --------- | ----------- | --------- | ----- |
| `PLAYER`  | 필수, 1~4만 허용 | 필수, 중복 불가 | 선택    |
| `COACH`   | 필수 아님       | 필수 아님     | 필수 아님 |
| `ANALYST` | 필수 아님       | 필수 아님     | 필수 아님 |

`PLAYER`가 학년 또는 등번호를 입력하지 않으면 회원가입 요청은 실패한다.

`PLAYER`의 학년이 1~4 범위를 벗어나면 회원가입 요청은 실패한다.

`PLAYER`의 등번호가 기존 선수와 중복되면 회원가입 요청은 실패한다.

등번호 중복 검증 대상은 다음과 같다.

* 삭제되지 않은 선수
* 승인 상태가 `PENDING` 또는 `APPROVED`인 선수

`REJECTED` 회원의 등번호는 재사용 가능하게 본다.

`COACH`, `ANALYST`가 학년, 등번호, 출신학교 값을 보내더라도 초기 구현에서는 저장하지 않고 null로 처리하는 것을 원칙으로 한다.

이유는 다음과 같다.

* 지도자와 분석관은 선수 식별 정보가 필요하지 않다.
* 프론트 조작으로 불필요한 선수 정보가 저장되는 것을 막는다.
* 회원 데이터 구조를 역할별로 명확하게 유지할 수 있다.

### 3.6 가입 승인 정책

지도자, 선수, 분석관 모두 관리자가 승인한 뒤 서비스를 사용할 수 있다.

---

## 4. 가입 승인 정책

### 4.1 기본 정책

신규 가입자는 즉시 서비스를 사용할 수 없고, 기본 승인 상태는 `PENDING`으로 저장한다.

승인 상태는 다음 3개를 사용한다.

| 승인 상태 | DB 값       | 의미                 | 로그인 가능 여부 |
| ----- | ---------- | ------------------ | --------- |
| 승인 대기 | `PENDING`  | 가입 신청 완료, 관리자 승인 전 | 불가        |
| 승인 완료 | `APPROVED` | 서비스 이용 가능 회원       | 가능        |
| 승인 거절 | `REJECTED` | 가입 거절 회원           | 불가        |

### 4.2 승인 주체

초기 버전에서는 `is_admin = true`인 관리자 회원만 가입 승인/거절을 처리할 수 있도록 설계한다.

이유는 다음과 같다.

* 하나의 팀이 사용하는 플랫폼이라도, 아무나 가입 즉시 접근하면 팀 영상과 선수 정보가 노출될 수 있다.
* 선수 개인 영상은 민감한 피드백 데이터이므로 가입 승인 절차가 필요하다.
* 지도자와 분석관 계정도 영상 관리 권한을 가질 수 있으므로 운영자 승인 없이 생성되면 위험하다.

### 4.3 최초 관리자 계정

최초 관리자 계정은 일반 회원가입 API로 만들지 않는다.

권장 방식은 다음 중 하나다.

1. 개발 초기에는 DB에 직접 1개 관리자 계정을 생성한다.
2. 운영 전에는 초기 관리자 생성용 seed 또는 별도 내부 API를 만든다.

초기 개발 단계에서는 1번 방식이 단순하다.

### 4.4 단일 승인/거절

관리자는 승인 대기 회원 한 명을 선택해서 승인 또는 거절할 수 있다.

```http
PATCH /api/admin/members/{memberId}/approval
```

허용되는 변경 상태는 다음 2개다.

* `APPROVED`
* `REJECTED`

`PENDING`으로 되돌리는 처리는 초기 버전에서는 허용하지 않는다.

### 4.5 일괄 승인/거절

관리자는 승인 대기 목록에서 여러 회원을 체크박스로 선택한 뒤 한 번에 승인하거나 거절할 수 있어야 한다.

```http
PATCH /api/admin/members/bulk-approval
```

요청 예시는 다음과 같다.

```json
{
  "memberIds": [2, 3, 4],
  "approvalStatus": "APPROVED"
}
```

처리 정책은 다음과 같다.

* 관리자만 사용할 수 있다.
* 승인 상태는 `APPROVED`, `REJECTED`만 허용한다.
* 선택 회원 목록이 비어 있으면 실패한다.
* 요청한 회원 중 존재하지 않거나 삭제된 회원이 있으면 전체 요청을 실패한다.
* 관리자 계정은 일괄 승인/거절 대상이 될 수 없다.
* 하나라도 문제가 있으면 전체 요청을 실패한다.

## 5. 로그인 정책

### 5.1 로그인 조건

로그인은 다음 조건을 모두 만족해야 성공한다.

* 로그인 ID가 존재해야 한다.
* 삭제된 회원이 아니어야 한다.
* 비밀번호가 일치해야 한다.
* 승인 상태가 `APPROVED`여야 한다.

하나라도 만족하지 않으면 로그인은 실패한다.

### 5.2 로그인 성공 시 처리

로그인 성공 시 다음 처리를 한다.

1. 회원 정보를 조회한다.
2. 비밀번호를 BCrypt로 검증한다.
3. 승인 상태를 확인한다.
4. `last_login_at`을 현재 시간으로 갱신한다.
5. JWT Access Token을 발급한다.
6. 프론트에는 토큰과 최소 회원 정보를 응답한다.

### 5.3 로그인 실패 응답 정책

보안상 로그인 실패 원인을 너무 자세히 노출하지 않는다.

로그인 ID가 없는 경우와 비밀번호가 틀린 경우는 모두 다음처럼 응답한다.

```text
아이디 또는 비밀번호가 올바르지 않습니다.
```

다만 승인 대기/거절 상태는 사용자가 자신의 가입 상태를 알아야 하므로 분리해서 안내할 수 있다.

```text
관리자 승인 후 로그인이 가능합니다.
```

```text
가입 승인이 거절된 계정입니다.
```

## 6. 비밀번호 암호화 방식

### 6.1 암호화 방식

비밀번호는 Spring Security의 `BCryptPasswordEncoder`를 사용해서 암호화한다.

DB에는 원문 비밀번호를 저장하지 않고 BCrypt 해시 값만 저장한다.

### 6.2 저장 방식

회원가입 요청 비밀번호는 다음 흐름으로 처리한다.

1. 사용자가 비밀번호를 입력한다.
2. 서버에서 비밀번호 형식을 검증한다.
3. `BCryptPasswordEncoder.encode()`로 암호화한다.
4. 암호화된 값을 `member.password`에 저장한다.

### 6.3 검증 방식

로그인 시에는 원문 비밀번호를 복호화하지 않는다.

다음 방식으로 검증한다.

```java
passwordEncoder.matches(rawPassword, encodedPassword)
```

### 6.4 비밀번호 정책

초기 비밀번호 정책은 다음을 권장한다.

* 최소 8자 이상
* 영문, 숫자 조합 권장
* 공백만 있는 비밀번호 금지
* 로그인 ID와 동일한 비밀번호 금지

운영 서비스 전에는 특수문자 포함, 실패 횟수 제한, 비밀번호 재설정 기능을 추가 검토한다.

## 7. JWT 사용 방식

### 7.1 기본 방식

초기 구현에서는 JWT Access Token 기반 인증을 사용한다.

프론트는 로그인 성공 후 발급받은 Access Token을 저장하고, 이후 API 요청마다 다음 헤더를 포함한다.

```http
Authorization: Bearer {accessToken}
```

### 7.2 Access Token 포함 정보

JWT에는 권한 검증에 필요한 최소 정보만 넣는다.

권장 Claims는 다음과 같다.

| Claim       | 설명                           |
| ----------- | ---------------------------- |
| `sub`       | 회원 ID                        |
| `loginId`   | 로그인 ID                       |
| `role`      | `COACH`, `PLAYER`, `ANALYST` |
| `isAdmin`   | 관리자 여부                       |
| `isCaptain` | 주장 여부                        |

`password`, `phone` 같은 민감 정보는 JWT에 넣지 않는다.

### 7.3 토큰 만료 시간

초기 개발 단계에서는 Access Token 만료 시간을 2시간으로 설정한다.

이유는 다음과 같다.

* 선수는 모바일에서 짧게 조회하는 경우가 많다.
* 지도자는 PC에서 영상 편집을 오래 할 수 있으므로 너무 짧으면 불편하다.
* 너무 긴 토큰은 탈취 시 위험하다.

운영 전에는 Refresh Token 도입을 검토한다.

### 7.4 Refresh Token 정책

초기 MVP에서는 구현 범위를 줄이기 위해 Refresh Token을 필수 구현에서 제외한다.

다만 실제 운영 서비스에서는 다음 방식으로 확장하는 것이 좋다.

* Refresh Token은 DB 또는 Redis에 저장한다.
* 브라우저 저장 시 HttpOnly Cookie 사용을 검토한다.
* 로그아웃 시 Refresh Token을 폐기한다.
* 토큰 재발급 API를 별도로 둔다.

현재 DB에는 Refresh Token 저장 테이블이 없으므로, Refresh Token을 바로 구현하려면 별도 테이블 설계가 필요하다.

### 7.5 JWT 검증 위치

JWT 검증은 Spring Security Filter에서 처리한다.

검증 흐름은 다음과 같다.

1. 요청 헤더에서 `Authorization` 값을 읽는다.
2. `Bearer` 토큰인지 확인한다.
3. JWT 서명과 만료 시간을 검증한다.
4. 회원 ID를 추출한다.
5. 회원이 삭제되지 않았는지 확인한다.
6. 승인 상태가 `APPROVED`인지 확인한다.
7. Spring Security 인증 객체를 생성한다.

## 8. API 흐름

### 8.1 회원가입 API

```http
POST /api/auth/sign-up
```

#### 선수 회원가입 요청 예시

```json
{
  "loginId": "player10",
  "password": "password123",
  "name": "홍길동",
  "phone": "01012345678",
  "memberRole": "PLAYER",
  "grade": 2,
  "almaMater": "서울중학교",
  "uniformNumber": 10
}
```

#### 지도자 회원가입 요청 예시

```json
{
  "loginId": "coach01",
  "password": "password123",
  "name": "김지도",
  "phone": "01011112222",
  "memberRole": "COACH"
}
```

#### 분석관 회원가입 요청 예시

```json
{
  "loginId": "analyst01",
  "password": "password123",
  "name": "박분석",
  "phone": "01033334444",
  "memberRole": "ANALYST"
}
```

#### 처리 흐름

1. 요청 값이 비어 있는지 검증한다.
2. `loginId` 중복 여부를 확인한다.
3. `memberRole`이 가입 가능한 역할인지 확인한다.
4. 역할이 `PLAYER`이면 학년과 등번호가 입력되었는지 확인한다.
5. 역할이 `PLAYER`이면 학년이 1~4 사이인지 확인한다.
6. 역할이 `PLAYER`이면 등번호가 중복되지 않았는지 확인한다.
7. 역할이 `COACH` 또는 `ANALYST`이면 선수 전용 입력값은 저장하지 않는다.
8. 비밀번호를 BCrypt로 암호화한다.
9. 회원을 `PENDING` 상태로 저장한다.
10. 가입 신청 완료 응답을 반환한다.

#### 응답 예시

```json
{
  "memberId": 1,
  "approvalStatus": "PENDING",
  "message": "회원가입 신청이 완료되었습니다. 관리자 승인 후 로그인할 수 있습니다."
}
```

## 10. DB 설계 방향

### 10.1 현재 사용할 회원 테이블 주요 컬럼

현재 회원가입/로그인 기능에서 우선 사용할 컬럼은 다음과 같다.

| 컬럼                | 사용 목적           |
| ----------------- | --------------- |
| `id`              | 회원 PK           |
| `login_id`        | 로그인 ID          |
| `password`        | BCrypt 암호화 비밀번호 |
| `phone`           | 휴대폰 번호          |
| `name`            | 회원 이름           |
| `grade`           | 선수 학년           |
| `alma_mater`      | 선수 출신학교         |
| `u_number`        | 선수 등번호          |
| `last_login_at`   | 마지막 로그인 시간      |
| `member_role`     | 회원 역할           |
| `is_captain`      | 주장 여부           |
| `is_admin`        | 관리자 여부          |
| `approval_status` | 가입 승인 상태        |
| `is_deleted`      | 소프트 삭제 여부       |
| `created_at`      | 생성일시            |
| `updated_at`      | 수정일시            |

### 10.2 역할별 DB 저장 정책

역할별 회원 정보 저장 정책은 다음과 같다.

| 역할        | `grade` | `alma_mater` | `u_number` |
| --------- | ------- | ------------ | ---------- |
| `PLAYER`  | 저장      | 입력 시 저장      | 저장         |
| `COACH`   | null 저장 | null 저장      | null 저장    |
| `ANALYST` | null 저장 | null 저장      | null 저장    |

선수만 학년과 등번호를 필수로 사용한다.

지도자와 분석관은 해당 값이 필요 없으므로 저장하지 않는다.

### 10.3 추가 검토가 필요한 컬럼/테이블

초기 회원가입/로그인 구현에서는 필수는 아니지만, 운영 서비스 전에는 다음을 검토해야 한다.

* Refresh Token 저장 테이블
* 로그인 실패 횟수 컬럼
* 비밀번호 변경일 컬럼
* 임시 비밀번호 또는 비밀번호 재설정 토큰 테이블
* 휴대폰 인증 또는 이메일 인증 정보

## 12. 주의사항

* 비밀번호 원문은 DB, 로그, 응답에 절대 남기지 않는다.
* JWT Secret Key는 코드에 직접 작성하지 않고 환경변수 또는 설정 파일로 분리한다.
* `application-secrets.properties`는 GitHub에 커밋하지 않는다.
* 회원가입 직후 자동 로그인은 하지 않는다. 승인 완료 전 접근을 막기 위해서다.
* 프론트에서 버튼을 숨겨도 백엔드 권한 검증은 반드시 필요하다.
* 역할별 입력 UI는 사용자 편의를 위한 것이고, 백엔드에서도 반드시 다시 검증해야 한다.
* 선수는 학년과 등번호가 반드시 필요하다.
* 선수 학년은 1~4까지만 허용한다.
* 선수 등번호는 기존 선수와 중복될 수 없다.
* 지도자와 분석관은 학년, 출신학교, 등번호를 저장하지 않는다.
* 선수 개인 영상 접근 제어는 로그인 기능 위에서 동작하므로, 인증 구조를 허술하게 만들면 전체 영상 보안이 무너진다.
* 현재 DB에는 팀 테이블이 없으므로, 초기 버전은 단일 팀 서비스 기준으로 회원을 관리한다.
* 추후 여러 팀을 받을 가능성이 생기면 회원과 팀 소속 구조를 다시 설계해야 한다.
