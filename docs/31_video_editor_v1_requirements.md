# 31. 경기 영상 편집기 v1 요구사항

## 1. 문서 목적

이 문서는 축구팀 운영 및 경기 영상분석 플랫폼의 핵심 기능인 경기 영상 편집기 v1의 최종 요구사항을 정의한다.

이 기능은 학습용 데모가 아니라 실제 축구팀에 판매하고 운영할 서비스를 기준으로 설계한다.

주요 목표는 다음과 같다.

- 지도자와 분석관이 PC에서 경기 장면을 빠르게 찾고 분석 클립을 제작한다.
- 팀 분석 클립과 선수 개인 분석 클립을 하나의 경기 영상 작업공간에서 제작한다.
- 드로잉, 텍스트, 화면 정지, 줌 효과를 최종 MP4에 합성한다.
- 선수는 모바일에서 완성된 분석 결과를 안정적으로 시청한다.
- 렌더링 장애, 서버 재시작, 중복 편집, 파일 삭제 상황에서도 데이터 정합성을 유지한다.

## 3. 사용자 역할과 권한

### COACH

영상 도메인의 모든 관리 권한을 가진다.

- 경기 원본 영상 등록·조회·수정·삭제·복구
- 팀 분석 클립 등록·조회·수정·삭제
- 선수 개인 분석 클립 등록·조회·수정·삭제
- 드로잉·텍스트·정지·줌 편집
- DRAFT 및 복구 스냅샷 사용
- 개별·일괄 렌더링
- FAILED 재시도
- 편집 잠금 강제 해제
- 전체 분석 클립 조회

### ANALYST

스케줄·공지사항 관리 권한은 제한하지만 영상 도메인에서는 COACH와 동일한 전체 권한을 가진다.

- 경기 원본 영상 등록·조회·수정·삭제·복구
- 팀 분석 클립 등록·조회·수정·삭제
- 선수 개인 분석 클립 등록·조회·수정·삭제
- 드로잉·텍스트·정지·줌 편집
- DRAFT 및 복구 스냅샷 사용
- 개별·일괄 렌더링
- FAILED 재시도
- 편집 잠금 강제 해제

### PLAYER

- 접근 가능한 경기 원본 영상 조회
- READY 팀 분석 클립 조회
- 본인의 READY 선수 개인 분석 클립 조회
- 편집 기능 접근 불가
- DRAFT, QUEUED, PROCESSING, FAILED 조회 불가

### 공통 보안 원칙

- 권한 검증은 반드시 백엔드에서 처리한다.
- 프론트 권한 분기는 사용자 경험 보조 용도다.
- `isAdmin`만으로 영상 관리 권한을 부여하지 않는다.

## 4. 편집 화면 기준

### 지원 환경

- 편집기는 PC 전용이다.
- v1 공식 편집 지원 브라우저는 최신 Chrome과 Edge를 우선한다.
- 선수용 결과 영상은 최신 모바일 Chrome, Safari, Edge에서 재생할 수 있어야 한다.
- 모바일과 태블릿에서는 편집 버튼을 노출하지 않고 결과 조회만 제공한다.

### 작업공간

하나의 경기 원본 영상 상세 화면에서 다음 작업을 모두 수행한다.

- 원본 영상 재생
- 현재 경기의 클립 목록 확인
- 새 팀 분석 클립 생성
- 새 선수 개인 분석 클립 생성
- 기존 DRAFT 또는 FAILED 편집
- 개별·일괄 렌더링
- 렌더링 상태 확인

현재 경기의 클립은 다음 상태 그룹으로 구분한다.

- 작업 중: DRAFT
- 생성 대기: QUEUED
- 생성 중: PROCESSING
- 완료: READY
- 실패: FAILED

글로벌 팀·선수 클립 관리 목록에서도 모든 관리 상태를 표시한다. DRAFT 클릭 시 해당 경기 영상 편집기로 이동한다.

## 5. 클립 생성 기본 정책

### 공통

- 하나의 클립은 하나의 연속 구간만 가진다.
- 비연속 장면은 각각 별도 클립으로 생성한다.
- 원본 구간 최소 길이는 1초다.
- 원본 구간 최대 길이는 15분이다.
- 화면 정지로 늘어난 시간을 포함한 최종 결과 영상도 최대 15분을 넘을 수 없다.
- 프론트와 백엔드가 모두 길이를 검증한다.
- 새 클립의 최초 상태는 DRAFT다.
- 사용자가 상태를 직접 선택하지 않는다.

### 팀 분석 클립 필수 입력

- 분석 대상 팀
- 제목
- 클립 유형

선택 입력:

- 지도자 코멘트

기존 클립 유형을 유지한다.

```text
HIGHLIGHT
ATTACK
DEFENSE
GOAL
CONCEDED
OFFSIDE
SETPIECE
ETC
```

### 선수 개인 분석 클립 필수 입력

- 대상 선수
- 제목
- 클립 유형

선택 입력:

- 지도자 코멘트

기존 클립 유형을 유지한다.

```text
PLAYER_GOOD
PLAYER_MISTAKE
SHOOTING
PASS
DRIBBLE
DEFENSE
POSITIONING
PRESSING
OFF_THE_BALL
ETC
```

## 6. 구간 선택과 시간 정밀도

### 구간 선택 방식

- I 버튼 또는 단축키로 시작점 설정
- O 버튼 또는 단축키로 종료점 설정
- 타임라인 핸들 조절
- 시작·종료 시간 직접 입력
- 선택 구간 반복 재생
- 선택 구간 길이 표시
- 렌더링 전 미리보기

### 이동 단위

- 일반 이동: 1초
- 큰 이동: 5초
- 세밀 이동: 원본 영상의 한 프레임

### 저장 단위

- 기존 초 단위 클립 시간은 밀리초 단위로 변경한다.
- 클립과 드로잉·효과 시간은 밀리초 정밀도로 저장한다.
- 원본 영상 FPS는 정수로 단순화하지 않고 분자·분모 또는 동일한 정확도를 보장하는 구조로 저장한다.

### 가변 프레임률 영상

- CFR 영상은 저장된 FPS 기준으로 한 프레임 이동을 제공한다.
- VFR 영상은 브라우저가 디코딩할 수 있는 가장 가까운 프레임으로 이동한다.
- v1에서는 전체 프레임 타임스탬프 인덱스를 생성하지 않는다.
- VFR에서 완전한 프레임 정확도가 보장되지 않는 경우 사용자에게 제한 안내를 표시한다.

## 7. 분석 요소

### 드로잉

- 선
- 화살표
- 원
- 박스
- 영역
- 텍스트
- 특정 선수 강조
- 패스 라인
- 수비 라인
- 압박 구역

좌표는 영상 원본 픽셀에 고정하지 않고 영상 표시 영역 기준 정규화 좌표로 저장한다. 레터박스 영역은 좌표 기준에서 제외한다.

드로잉은 서로 겹칠 수 있다.

### 텍스트

- HTML을 허용하지 않는다.
- 일반 문자열만 저장하고 렌더링한다.
- 최대 길이와 전체 편집 데이터 크기는 외부 설정으로 제한한다.

### 화면 정지

- 원본 영상의 특정 시점 프레임을 일정 시간 정지한다.
- 기본 유지 시간은 3초다.
- 최소 1초, 최대 10초다.
- 정지 구간의 오디오는 무음 처리한다.
- 정지 종료 후 원본 오디오와 영상을 이어서 재생한다.
- 동일 시점에 중복 정지 효과를 등록하지 않는다.

### 줌

- 특정 구간에 하나의 관심 영역과 배율을 지정한다.
- 기본 배율은 1.5배다.
- 허용 범위는 1.2배 이상 3배 이하다.
- 줌 구간끼리는 겹치지 않도록 검증한다.
- 줌과 드로잉은 함께 사용할 수 있다.
- v1에서는 다중 키프레임 카메라 이동과 복잡한 팬 애니메이션을 제공하지 않는다.

### 편집 데이터 제한

운영 장애와 과도한 렌더링 명령 생성을 막기 위해 다음 제한은 외부 설정으로 관리한다.

- 클립당 전체 편집 요소 수
- 텍스트 최대 길이
- 편집 요청 JSON 최대 크기
- 화면 정지 개수
- 줌 효과 개수

초기 기본값은 전체 편집 요소 300개, 요청 데이터 2MB 이하로 시작하고 실제 사용 테스트 후 조정한다.

## 8. 자동 저장과 복구

### 자동 저장

- 변경 후 1~2초 debounce 저장
- 변경이 계속되는 경우 30초마다 안전 저장
- 정상 화면 종료 시 최종 저장 시도
- 저장 중에는 동일 요청 중복 전송을 막는다.

### 브라우저 로컬 복구

- 서버 저장 실패 시 편집 데이터를 IndexedDB에 보관한다.
- 사용자, 클립, 편집 버전 단위로 구분한다.
- 서버에 동일하거나 더 최신 버전이 정상 저장되면 로컬 미동기화 데이터를 정리한다.
- READY 전환 또는 클립 삭제 시 해당 로컬 복구 데이터를 삭제한다.

### 복구 스냅샷

다음 시점에 서버 복구 스냅샷을 생성한다.

- DRAFT 편집 중 10분마다
- 사용자 수동 저장
- 렌더링 요청 직전
- 강제 잠금 해제 직전
- 이전 스냅샷 복원 직전

보관 제한:

- 클립당 최신 스냅샷 최대 20개
- 한도 초과 시 가장 오래된 자동 스냅샷부터 삭제
- READY 완료 시 모든 복구 스냅샷 삭제
- 최종 편집 원본 데이터는 READY 이후에도 유지

스냅샷 복원은 현재 편집 잠금을 가진 사용자만 수행할 수 있다. 복원 직전에 현재 상태를 별도 스냅샷으로 보관하고 편집 버전을 증가시킨다.

## 9. 편집 잠금과 충돌 방지

### 잠금 단위

- 경기 전체가 아니라 클립 단위로 잠근다.
- 서로 다른 클립은 동시에 편집할 수 있다.
- 같은 클립은 한 명만 편집하고 다른 사용자는 읽기 전용으로 접근한다.

### 잠금 유지

- heartbeat 주기: 30초
- heartbeat가 10분 동안 없으면 잠금 만료
- 정상 종료 시 즉시 잠금 해제 시도
- 비정상 종료는 만료 정책으로 복구

### 잠금 토큰과 편집 버전

- 잠금 획득 시 예측 불가능한 잠금 토큰을 발급한다.
- 저장 요청은 잠금 토큰과 편집 버전을 함께 전달한다.
- 오래된 버전 또는 무효 잠금 토큰의 저장은 `409 Conflict`로 차단한다.
- 잠금 구현은 서비스 인터페이스로 분리하고 초기에는 MySQL을 사용한다.
- 향후 Redis 잠금으로 교체할 수 있어야 한다.

### 강제 잠금 해제

- COACH와 ANALYST 모두 가능하다.
- 강제 해제 전에 현재 서버 편집 상태를 스냅샷으로 저장한다.
- 기존 편집자의 토큰은 즉시 무효화한다.
- 기존 편집 화면은 다음 저장 시 읽기 전용으로 전환하고 충돌 안내를 표시한다.

### 동일 사용자의 여러 탭

- 동일 사용자가 다른 탭에서 같은 클립을 열면 기존 잠금을 자동 공유하지 않는다.
- 새 탭에서 명시적으로 이어받기를 선택하면 새 잠금 토큰을 발급하고 기존 탭 토큰을 무효화한다.

## 10. 클립 상태 정책

클립 상태는 다음 다섯 개를 사용한다.

```text
DRAFT
QUEUED
PROCESSING
READY
FAILED
```

기존 클립의 `UPLOADING` 상태는 신규 편집 흐름에서 사용하지 않는다.

### DRAFT

- 편집 및 자동 저장 가능
- 개별·일괄 렌더링 가능
- PLAYER 조회 불가
- 10일 동안 수정이 없으면 `장기 미작업` 표시
- 자동 삭제하지 않음
- 수동 삭제 시 즉시 영구 삭제

### QUEUED

- 렌더링 대기 상태
- 편집·삭제·재시도 불가
- PLAYER 조회 불가
- 60분 타임아웃 계산 대상 아님

사용자 문구:

```text
영상 생성 대기 중
```

### PROCESSING

- 실제 FFmpeg가 실행 중인 상태
- 편집·삭제·재시도 불가
- PLAYER 조회 불가
- 실제 시작 후 60분이 지나면 프로세스를 종료하고 FAILED 처리

사용자 문구:

```text
영상 생성 중
```

삭제 차단 문구:

```text
영상 생성 중인 클립은 삭제할 수 없습니다. 생성이 완료되거나 실패한 후 다시 시도해주세요.
```

### READY

- 완성된 MP4 재생 가능
- PLAYER 권한 범위에 따라 즉시 노출
- 별도 게시 상태는 v1에서 사용하지 않음
- 제목, 클립 유형, 코멘트만 직접 수정 가능
- 구간, 대상 선수, 드로잉, 정지, 줌은 직접 수정 불가
- 영상 내용을 변경하려면 보관된 최종 편집 데이터를 복사해 새 DRAFT 생성

### FAILED

- 편집 원본 데이터 유지
- 편집 계속하기, 다시 렌더링, 삭제 가능
- PLAYER 조회 불가
- 재시도 시 동일 클립 행을 사용한다.

상태 흐름:

```text
DRAFT → QUEUED → PROCESSING → READY
DRAFT → QUEUED → PROCESSING → FAILED
FAILED → QUEUED → PROCESSING → READY 또는 FAILED
```

---

## 11. 렌더링 요청 정책

### 개별 렌더링

- DRAFT 또는 FAILED만 요청 가능
- 최종 자동 저장 성공 확인
- 렌더링 직전 복구 스냅샷 생성
- 상태를 QUEUED로 변경
- 기존 실패 사유 초기화
- 편집 잠금 해제

### 일괄 렌더링

- DRAFT와 FAILED 선택 가능
- 최소 1개, 최대 20개
- QUEUED, PROCESSING, READY 선택 불가
- 각 클립을 독립적으로 검증하고 처리한다.
- 한 클립 실패가 다른 클립에 영향을 주지 않는다.
- 유효한 클립만 대기열에 등록한다.
- 요청 결과는 클립별 성공·제외 사유를 반환한다.

일괄 요청 내부 순서:

```text
created_at ASC, clip_id ASC
```

실제 전체 대기열은 렌더링 요청 시각 기준 FIFO다.

---

## 12. 렌더링 대기열

### 동시 실행

- 팀 분석 클립과 선수 개인 분석 클립이 하나의 공용 대기열을 사용한다.
- 실제 동시 렌더링 최대 개수는 2개다.
- 설정값으로 변경 가능해야 한다.
- 역할 또는 클립 종류에 따른 우선순위를 두지 않는다.

### 대기 순서

- 렌더링 요청 시각 오름차순
- 동일 시각은 클립 종류와 클립 ID를 이용해 항상 같은 순서를 보장한다.
- 렌더링 요청 시각은 마이크로초 정밀도를 지원하는 DB 타입을 우선한다.

### 구현 방향

- 별도 렌더링 작업 테이블은 v1에서 추가하지 않는다.
- 팀·선수 클립 테이블에 렌더링 요청 시각과 실제 시작 시각을 저장한다.
- DB 상태를 대기열의 기준 데이터로 사용한다.
- 스케줄러가 실행 가능 슬롯을 확인하고 다음 QUEUED 작업을 원자적으로 PROCESSING으로 점유한다.
- 현재 단일 서버 기준으로 구현하되 서비스 계층을 분리해 이후 전용 작업 큐로 이전할 수 있어야 한다.

### 서버 재시작

서버 시작 시:

1. 남아 있는 PROCESSING 클립 조회
2. 실패 사유를 SERVER_RESTART로 저장하고 FAILED 전환
3. 불완전한 임시 파일 정리
4. QUEUED 클립을 요청 순서대로 다시 대기열에 등록
5. 동시 실행 한도에 맞춰 렌더링 재개

자동으로 기존 PROCESSING 작업을 재실행하지 않는다.

사용자 문구:

```text
서버 재시작으로 클립 영상 생성이 중단되었습니다. 다시 렌더링해주세요.
```

---

## 13. 렌더링 실패와 타임아웃

### 실패 사유

```text
RENDER_ERROR
TIMEOUT
SERVER_RESTART
```

- FAILED 상태에서만 사용자 안내에 사용한다.
- 내부 예외 메시지, 서버 경로, FFmpeg 전체 명령어는 API에 노출하지 않는다.
- 재렌더링 요청 시 실패 사유를 초기화한다.

### 사용자 문구

```text
RENDER_ERROR: 렌더링 실패
TIMEOUT: 렌더링 시간 초과
SERVER_RESTART: 서버 재시작으로 중단
```

### 타임아웃

- 실제 PROCESSING 진입 시각부터 60분
- QUEUED 대기 시간은 포함하지 않음
- 제한 초과 시 FFmpeg 프로세스 종료
- 불완전한 임시 파일 삭제
- 편집 원본 데이터 유지
- FAILED 전환 및 TIMEOUT 저장

### 자동 재시도

- 렌더링 작업은 자동 재시도하지 않는다.
- 사용자가 편집 내용을 확인한 뒤 수동 재시도한다.

## 14. 렌더링 결과 규격

### 컨테이너와 코덱

```text
컨테이너: MP4
영상 코덱: H.264(libx264)
오디오 코덱: AAC
픽셀 포맷: yuv420p
```

### 화질

- CRF 기반 가변 비트레이트
- 초기 CRF 21
- preset medium
- 설정값으로 변경 가능

### 해상도

- 최대 1080p
- 원본이 1080p 이하면 원본 해상도 유지
- 원본보다 확대하지 않음
- 화면 비율 유지
- 16:9 강제 자르기 금지

### 프레임률

- 최대 60fps
- 원본보다 높이지 않음
- 프레임 보간하지 않음

### 오디오

- 원본에 오디오가 있으면 AAC 128kbps, 최대 2채널, 48kHz
- 원본에 오디오가 없으면 무음 트랙을 새로 만들지 않음
- 화면 정지 구간에는 무음 삽입

### 웹 재생

- faststart 적용
- 키프레임 간격은 약 2초
- 결과 파일은 Range 재생을 지원해야 한다.

### 설정 외부화

- CRF
- preset
- 최대 해상도
- 최대 FPS
- 오디오 비트레이트
- 동시 렌더링 수
- 렌더링 타임아웃
- 편집 요소 제한

## 15. 렌더링 결과 검증

READY 전환 전 다음을 확인한다.

- FFmpeg 정상 종료 코드
- 임시 MP4 파일 존재
- 파일 크기 0 초과
- ffprobe로 영상 스트림 확인
- H.264 영상 코덱 확인
- 출력 길이가 예상 범위와 일치하는지 확인
- 최대 해상도와 최대 FPS 정책 준수 확인
- 최종 저장 경로 이동 성공

검증이 모두 완료된 뒤 READY 상태와 파일 정보를 저장한다.

편집 원본 데이터와 결과 MP4의 연결을 추적할 수 있도록 다음 정보를 유지한다.

- 렌더링 요청 시각
- 렌더링 시작 시각
- 렌더링 완료 시각
- 결과 영상 길이
- 결과 해상도
- 결과 FPS
- 결과 파일 크기
- 적용한 렌더링 프로필 버전

렌더링 프로필 버전을 저장해 향후 규격 변경 후 기존 클립 재생성 여부를 판단할 수 있게 한다.

## 16. 썸네일

### 생성

- READY MP4의 시작 후 1초 지점에서 자동 생성
- 1초 프레임 실패 시 중간 지점
- 다시 실패하면 최초 디코딩 가능 프레임
- 모두 실패하면 기본 썸네일
- JPEG 사용
- 가로 최대 640px
- 화면 비율 유지
- 업스케일링하지 않음

### 실패 처리

- MP4가 정상이라면 READY 유지
- 기본 썸네일 표시
- 서버 로그 기록
- 사용자에게 렌더링 실패로 표시하지 않음

### 삭제

- 클립 영구 삭제 시 썸네일도 삭제 대상으로 등록한다.

## 18. READY 클립 수정과 복제

### 직접 수정 가능

- 제목
- 클립 유형
- 코멘트

### 직접 수정 불가

- 원본 경기 영상
- 시작·종료 구간
- 선수 개인 클립의 대상 선수
- 드로잉·텍스트
- 화면 정지
- 줌 효과
- 결과 MP4

영상 내용을 변경하려면 최종 편집 원본 데이터를 복사해 새 DRAFT를 생성한다. 기존 READY 클립은 유지하며 새 DRAFT가 READY가 된 뒤 사용자가 기존 클립을 별도로 삭제할 수 있다.

### 원본 영상이 삭제된 경우

- 원본이 소프트 삭제 상태면 먼저 원본을 복구해야 한다.
- 원본 파일이 영구 삭제됐으면 READY 기반 새 DRAFT 생성을 차단한다.
- 완성된 READY MP4를 다시 편집 원본으로 사용하지 않는다.

문구:

```text
원본 경기 영상이 삭제되어 이 클립을 기반으로 새 클립을 만들 수 없습니다.
```

## 19. 영상 접근과 스트리밍

### 기본 정책

경기 원본, 팀 분석 클립, 선수 개인 분석 클립과 썸네일은 운영 환경에서 `/uploads/**` 정적 공개를 사용하지 않는다.

DB에는 외부 공개 URL이 아니라 내부 저장 키 또는 상대 저장 경로를 저장한다. API 응답에 실제 서버 파일 경로를 노출하지 않는다.

### 재생 방식

현재 인증이 Bearer Token 기반이고 HTML video 요소는 Authorization 헤더를 직접 넣기 어렵기 때문에 다음 2단계 방식을 사용한다.

1. 프론트가 JWT로 재생 권한 발급 API 요청
2. 백엔드가 특정 사용자·특정 영상에만 유효한 단기 재생 토큰 발급
3. video 요소가 단기 토큰이 포함된 스트리밍 URL 사용
4. 스트리밍 API가 토큰, 만료시간, 대상 리소스, 사용자 권한을 검증
5. HTTP Range 요청에 `206 Partial Content`로 응답

### 단기 재생 토큰

- 기본 만료시간 10분
- 특정 리소스에만 사용 가능
- 다른 원본·클립으로 재사용 불가
- 변조 방지를 위한 서버 서명 적용
- DB에 영구 저장하지 않음
- 만료 후 필요한 경우 프론트가 새 토큰 요청
- URL과 로그에 장기 JWT를 포함하지 않음

### 권한

- 경기 원본: 기존 역할별 원본 조회 정책 적용
- 팀 클립: 팀 소속 사용자 및 COACH·ANALYST
- 선수 개인 클립: 대상 선수 본인 및 COACH·ANALYST
- 썸네일도 원본 영상과 동일한 권한 적용
- 삭제된 원본 영상은 파일이 남아 있어도 스트리밍 차단

### 응답

- `Content-Disposition: inline`
- Range 요청 지원
- 공개 캐시 금지
- 사용자별 private 캐시만 짧게 허용
- v1에서 별도 다운로드 버튼 제공하지 않음

이 방식은 DRM이 아니므로 화면 녹화와 사용자의 기술적 저장까지 완전히 막을 수는 없다.

## 20. 상태 갱신 UI

- 렌더링 상태는 초기 버전에서 3초 간격 폴링으로 갱신한다.
- 모든 대상이 READY 또는 FAILED가 되면 폴링을 중지한다.
- 브라우저 탭이 백그라운드면 폴링 주기를 늘린다.
- v1에서는 WebSocket과 SSE를 도입하지 않는다.
- 렌더링 완료 또는 실패 시 화면 배지와 사용자 안내를 갱신한다.

## 21. 클립 삭제 정책

### 직접 클립 삭제

팀 분석 클립과 선수 개인 분석 클립은 원본 경기 영상과 달리 복구 기간 없이 즉시 영구 삭제한다.

삭제 가능 상태:

- DRAFT
- FAILED
- READY

삭제 불가 상태:

- QUEUED
- PROCESSING

확인 문구:

```text
이 클립을 영구 삭제하시겠습니까?

삭제하면 클립 영상과 모든 관련 데이터가 즉시 영구 삭제되며 복구할 수 없습니다.
```

버튼:

```text
취소
영구 삭제
```

영향 데이터 개수는 표시하지 않는다.

### 클립 삭제 대상

- 클립 DB 행
- 완성 MP4
- 썸네일
- 최종 편집 원본 데이터
- 드로잉·텍스트
- 정지·줌 설정
- 북마크
- 선수 개인 클립 조회 기록
- 선수 기록 이벤트와 클립 연결 정보
- 복구 스냅샷
- 임시 파일

### 유지 대상

- 선수 기록 이벤트 자체
- 선수 경기 기록
- 선수 누적 통계

### 파일과 DB 정합성

외부 파일 삭제는 DB 트랜잭션에 포함할 수 없으므로 다음 구조를 사용한다.

1. 클립 관련 DB 데이터를 트랜잭션으로 삭제
2. 같은 트랜잭션에서 파일 정리 작업을 기록
3. 커밋 직후 MP4, 썸네일, 임시 파일 즉시 삭제 시도
4. 실패 시 재시도하고 운영 로그 기록

DB가 삭제된 이후 스트리밍 API에서 접근할 수 없으므로 파일 삭제 재시도 중에도 사용자에게는 영구 삭제된 상태로 처리된다.

## 22. 원본 경기 영상 삭제 정책

### 삭제 차단

해당 원본을 사용하는 팀·선수 클립 중 QUEUED 또는 PROCESSING이 하나라도 있으면 원본 삭제를 차단한다.

```text
이 경기 영상으로 생성 중인 클립이 있어 삭제할 수 없습니다. 클립 생성이 완료되거나 실패한 후 다시 시도해주세요.
```

실행 중 FFmpeg를 원본 삭제 요청으로 강제 종료하지 않는다.

### 삭제 영향 확인창

다음만 표시한다.

1. 원본 경기 영상 개수와 제목
2. 원본 북마크 총개수
3. 팀 분석 클립 총개수와 각 제목

미완성 선수 개인 클립 개수나 제목은 표시하지 않는다.

경고:

- 원본 영상은 10일 동안 복구 가능
- 원본 북마크와 팀 분석 클립은 복구되지 않음
- 팀 분석 클립 MP4는 즉시 영구 삭제
- 10일 후 원본 MP4가 영구 삭제되며 원본 복구 불가

### 삭제 시점 처리

- `game_video_upload.is_deleted = true`
- `deleted_at = 현재 시각`
- 원본 MP4는 10일 유지
- 원본 북마크는 소프트 삭제하고 복구하지 않음
- 연결된 팀 분석 클립은 소프트 삭제
- 팀 분석 클립 MP4는 즉시 물리 삭제 대상으로 등록
- 팀 클립 드로잉과 팀 클립 북마크는 소프트 삭제하고 복구하지 않음
- 선수 개인 READY 클립과 관련 데이터는 유지
- 선수 기록과 선수 기록 이벤트는 유지

원본 삭제로 소프트 삭제된 팀 분석 클립과 선수 기록 이벤트 연결 이력은 유지한다. 화면에는 다음을 표시한다.

```text
해당 클립이 삭제되었습니다
```

### 10일 후 정리

- 원본 MP4 영구 삭제
- 연결된 선수 개인 DRAFT와 FAILED 클립 영구 삭제
- 선수 개인 READY 클립 유지

- 물리 파일 삭제 실패 시 재시도 및 로그 기록

### 복구

삭제된 영상 버튼을 경기 영상 화면에 제공한다.

모달 표시 항목:

- 제목
- 삭제 일시
- 복구 또는 복구 불가 버튼

정책:

- 10개 단위 페이지네이션
- `deleted_at DESC`
- 검색과 필터 없음
- 삭제 이력 메타데이터는 영구 유지

## 24. API 흐름

정확한 URL은 기존 Controller 구조를 확인한 뒤 확정한다. 역할이 드러나는 관리 API와 선수 조회 API를 분리한다.

### 편집 작업

- 현재 경기의 관리용 전체 클립 상태 조회
- DRAFT 생성
- DRAFT 상세 조회
- 편집 잠금 획득·heartbeat·해제·강제 해제

- 클립 영구 삭제

### 렌더링 요청 응답

개별 렌더링:

- 정상 대기열 등록: `202 Accepted`
- 잘못된 상태, 활성 편집 잠금 충돌: `409 Conflict`

일괄 렌더링:

- 요청 형식이 정상이라면 클립별 결과를 포함해 `200 OK`
- 0개 또는 20개 초과: `400 Bad Request`
- 권한 없음: `403 Forbidden`
- 각 항목에 대기열 등록 또는 제외 사유 포함

### 스트리밍

- JWT 인증 기반 재생 토큰 발급 API
- 토큰 기반 원본 영상 스트리밍 API

- Range 요청 지원

## 26. 파일 저장과 보안

- 사용자 원본 파일명을 실제 저장 경로로 사용하지 않는다.
- UUID 등 서버 생성 저장명을 사용한다.

- FFmpeg 실행은 문자열 쉘 명령 조합이 아니라 인수 리스트 기반 프로세스 실행을 사용한다.

- 텍스트 렌더링은 별도 파일 또는 안전하게 이스케이프된 입력으로 처리한다.
- 임시 파일은 작업별 격리 디렉터리를 사용한다.
- 작업 완료, 실패, 타임아웃, 서버 재시작 시 정리한다.
